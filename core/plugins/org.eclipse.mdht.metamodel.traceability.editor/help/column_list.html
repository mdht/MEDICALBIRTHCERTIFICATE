Column Groups

Each column in a table view shows a model element, or one of its attributes. The columns have the cwnaming convention Prefix: Element, and are grouped according the source for the model (usually a file). The prefix in the name of the column indicates the source of the element.

The model sources are currently grouped/prefixed as follows:

Req:
The Information Requirements derived from the NEAR tooling, exported as a ReqIF file
SCS:
The Structured Content Specification, as represented by an XML Expanded Structure File. The content of these columns may also be derived from other Logical Model XML files that are referenced from the SCS Expanded Structure.
Doc:
The DocBook XML file containing the hand-edited Requirement Id to SCS Path mapping table.
Dts:
The "details file" for a given DCM, or re-usable data component. This file is the source of a re-used <Mapping> element that the Logical Model Importer used to create an equivalent CDA UML class, which can found in the UML: prefixed columns.
UML:
The MDHT CDA UML model elements that represent the Implementation Reference CDA constraints, constructed by the Logical Model Importer, or added by CDA modellers after LMI import. Some columns prefixed as UML: do not appear to be UML related, but are in fact deduced from the UML structure, rather than being directly derived from the generated artefacts of the MDHT document and code generation. e.g. CDA IR Section, which is deduced before the actual PDF document is generated.

The exceptions to this column naming format of colon-separated prefixes are Comments columns that allow comments about traces between model sources to be added by Trace Matrix users.

Special Columns

Project: This column is typcially only shown when doing trace differencing, and indicates which rows in the Trace Matrix table come from which Eclipse Project.

Error: This is a special column that aggregates any errors that are detected in correlations or mappings between the model sources. These may be LMI Import errors for SCS elements that cannot be inserted correctly into the UML CDA structure based on their re-usable mappings, or they may be failed correlations between columns that are meant to match.

Selecting any row containing a populated Error column cell will cause the "EMF Tree" view, if present, to highlight the corresponding trace where the error occurs.

Comment Columns

Req/SCS Comments. This column allows the annotation of any mapping between an Information Requirement and an SCS Path (as deduced from the DocBook Requirement/SCS Mapping). Currently these comments are only stored with the mapping in a given project, and are not propagated to new drafts of the Logical or Implementation models.

SCS/Dts Comments. This column allows the annotation of any mapping between an SCS Data Element and the "details" XML file in which a <Mapping> element describes how it should be represented in CDA. Currently these comments are only stored with the mapping in a given project, and are not propagated to new drafts of the Logical or Implementation models.

Information Requirements Columns

Req: No. The Information Requirement number from NEAR.

Req: Section. A grouping of information requirements in NEAR realised as a document section contiaining related requirements in the published Information Requirements Specification.

Req: EA ID. The Enterprise Architect identifier for the Information Requirement number in NEAR.

Req: Detail. The Information Requirement detailed description text.

Req: Description. The Information Requirement high-level descrition.

Req: Rationale. An Information Requirement rationale.

SCS Columns

SCS: Data Component. The name of the SCS Expanded Structure file node.

SCS: Cardinality. The cardinality of the SCS data component as defined in the logical model derived from the Expanded Structure File.

DocBook Mapping Table Columns

Doc: Logical Path. The text of any right-hand-side SCS Path that is asserted to implement an Information Requirement in the DocBook Requirments to SCS Mapping Table. The Requirement Number is not shown as a "Doc" column, as it is correlated with the Req: No column from the Requirements, and is shown only once under this heading in the table.

Details File Columns

Dts: Mapping. The CDA type which is specified as the mapping for an SCS data component in a "Details" XML file in one of its <Mapping> elements. This information is used by the LMI to create the UML structure which is deduced from the type information in the context of the SCS Tree, and the CDA structure that was cteated for the parents of this SCS component.

Dts: fixed_value. The fixed value specified for an SCS data component in an XPath expression in a <Mapping> element of a "Details" XML file. This information is used by the LMI to create an equivalent UML attribute default value.

UML Columns

UML: Type. The CDA Base Type for the MDHT UML Element, as deduced by the LMI from the <Mapping> elements of the logical model /ML Details files.

UML: Element. The specific MDHT UML Class or Association that uses the CDA type to specify the constraint equivalent to the SCS Data Component shown on the same row.

UML: Multiplicity. The multiplicity of the UML Element.

UML: Value. The default value of a UML Element which has a fixed value specified in the SCS.

UML: Conformance. The text of the conformance statement for the constraint generated by MDHT that will appear in the Implemenatation Reference Document.

UML: CDA IR Section. The deduced section number of the constraint that will be generated in the Implementation Reference Document.

UML: Java. The Java method generated by MDHT that will validate the constraint for the correlated UML Element.

Experimental Columns not in production tool views

Req: CCA Comments. Comments on Information Requirements by CCA Staff for visibility to other roles in the specification and publication processes.

Req: Clinical Safety Comments. Comments on Information Requirments by Clinical Safety Staff for visibility to other roles in the specification and publication processes.

Req: Clinical Safety Issue. Issue cross references for Clinical Safety Staff, typically relating to Hazards or Jira tickets.

SCS: DCM. The name of a DCM Structure File from which the definition of an SCS data component is derived.

SCS: Archetype. The name of an Archetype XML File from which the definition of an SCS data component is derived.
